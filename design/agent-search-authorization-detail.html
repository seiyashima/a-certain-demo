Detailed Design

Agent Search 認可判定の詳細設計

検索時に、どの仕組みを使ってどこで認可判定するかを整理する。一次判定は Agent Search、 コンプライアンス/GDPR 区分のみ Cloud Run で二次判定する。

1. 判定アーキテクチャ

層 役割 使う情報
Agent Search（一次判定） 検索対象を抽出し、ACL メタデータで閲覧可否を絞り込む ユーザの SSO コンテキスト、ドキュメントの ACL メタデータ
Cloud Run Policy Guard（二次判定） 高機密区分のみ追加ポリシーを適用して過剰公開を防止する 一次判定後の候補、機密区分フラグ、業務ポリシー
Gemini Enterprise（回答生成） 最終的に許可された結果だけを根拠に回答を生成する 最終判定済み検索結果、引用情報

1-1. 何のために突合するか（目的の明確化）

JWT クレームと ACL メタデータを突合する目的は、 「認証済みユーザに対して、閲覧許可された文書だけを返す」ことを検索時に機械的に実現するためである。 取得時（ETL）に高権限で収集したデータを、そのまま無差別に返さないための最終制御点がこの突合である。

目的 防ぐリスク 期待効果
認可一貫性の担保 「ログイン済み=全件閲覧可」という誤判定 許可された対象だけを返却し、過剰開示を防止する
最小権限の実装 権限過大な横断検索 ユーザ属性と ACL の一致条件で表示範囲を最小化する
監査可能性の確保 なぜ見えた/見えないか説明できない状態 突合キーと判定結果をログ化し、事後検証を可能にする

2. Agent Search で使う判定キー

ETL で格納した ACL メタデータと、ユーザの属性を突合する。代表キーは次のとおり。

ユーザ属性（例） sub, groups, department, region, position, manager_id
ACL メタデータ（例） allowed_groups, allowed_departments, allowed_regions, allowed_positions, allowed_manager_ids, allowed_user_subs
追加分類 classification（例: standard, compliance, gdpr）

3. 判定プロセス

  1. ユーザは Okta SSO 後のコンテキストで検索要求を送る
  2. Agent Search がデータストアから候補を取得する
  3. Agent Search が ACL メタデータで一次判定する
  4. 候補に compliance/gdpr 区分が含まれる場合のみ Cloud Run が二次判定する
  5. 最終的に許可された結果のみ Gemini Enterprise が回答生成に使う

4. 突合ルール（具体）

一次判定は次の順で評価する。複数条件は AND で扱い、1 つでも不一致なら deny とする。

  1. トークン妥当性確認（iss, aud, exp など）
  2. 主体特定（sub を正規化して内部 principal_id に変換）
  3. 属性突合（groups / department / region / position / manager_id）
  4. 個別許可突合（allowed_user_subs に sub が含まれるか）
  5. 区分判定（classification が compliance/gdpr の場合のみ二次判定へ）
主キー sub（必要に応じて issuer と組み合わせて主体を一意化）
補助キー groups, department, region, position, manager_id
不採用キー email（運用上変更されうるため、主体の主キーにしない）
評価ポリシー deny by default / fail closed

5. フェイルセーフ方針

  • ACL メタデータ不足時は deny（非表示）をデフォルトにする
  • classification 不明時は高機密扱いとして二次判定へ回す
  • 二次判定でエラー時は結果を返さない（fail closed）

6. 設計根拠（エビデンス）

  • JWT はクレームを安全に運搬する標準形式であり、iss/sub/aud/exp などの検証を前提に利用する: RFC 7519 (JSON Web Token)
  • OIDC では ID Token による主体識別を行い、sub は主体識別子として扱う。識別の安定性は iss+sub の組み合わせが基準: OpenID Connect Core 1.0
  • Entra / Microsoft Graph の権限モデルは delegated と app-only を分離し、最小権限付与を推奨: Overview of Microsoft Graph permissions / Microsoft Graph permissions reference
  • Agent Search ではメタデータフィルタ（filter）により検索結果を条件付きで絞り込める: Filter search by metadata
  • WIF はサービスアカウント鍵の代わりに連携 ID と短命トークンを使う方式であり、鍵管理リスクを下げる: Workload Identity Federation
