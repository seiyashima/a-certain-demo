Connector Architecture

連携アーキテクチャ設計

Gemini Enterprise Agent Platform から呼ばれる Cloud Run アプリの公開可能な設計判断を整理したドキュメントです。

1. 設計ゴール

  • 複数ソースを統一的な検索体験で束ねる
  • 元システムの権限を維持し、見える情報だけ返す
  • 短命認証と監査ログを前提に運用する
  • Cloud Run 上で再現可能な構成にする
  • Search API / Ops UI / Connector Worker の責務を分離する

2. 主要コンポーネント

UI layer 検索画面と稼働確認画面を提供する
API layer /healthz, /api/runtime, /api/chat を提供する
Connector layer 各ソースの取得ロジックを個別実装で分離する
Identity layer Okta などの外部IdPと GCP の短命認証を仲介する

3. 認証・権限の考え方

ユーザーは既存SSOで認証し、そのアイデンティティをそのままソースアクセスの判断材料として扱う。 サービス側は長期資格情報を保持せず、必要な時だけ短命トークンを取得する。

  • ユーザー認証は既存IdPを起点に統一する
  • Connector はソース固有の権限モデルを保持する
  • 検索時には ACL フィルタで結果を絞り込む
  • ドキュメント更新時は再インデックスまたは削除を行う
  • Cloud Run サービスは責務別に分割し、独立して運用する
クローリング時は Cloud Run がシステム連携用の高権限でデータと ACL を取得し、検索用データストアへ整形して格納する。 ユーザ検索時は、この格納済みデータと ACL メタデータを基準に参照可否を判定する。 標準判定は Agent Search 側で行い、コンプライアンス/GDPR などの高機密区分のみ Cloud Run 側で二重チェックする。

3-1. 5システムの認証方式一覧

システム identity_provider 認証方式 利用プロセス
ServiceNow okta OAuth 2.0 Client Credentials（Okta） Cloud Run が Okta token endpoint で access token を取得し、Bearer token で ServiceNow API を呼び出す
Workday entra_id OAuth 2.0 Client Credentials（Entra ID） Cloud Run が WIF（Federated Credential）前提で Entra ID token endpoint から access token を取得し、Bearer token で Workday API を呼び出す
Compliance System okta_ldap_agent Okta LDAP Agent 経由の資格情報連携 Cloud Run が Secret Manager から LDAP 用 username/password を取得し、ヘッダー経由で Compliance API へ連携する
SharePoint entra_id OAuth 2.0 Client Credentials（Entra ID / Microsoft Graph） Cloud Run が WIF（Federated Credential）前提で Entra ID token endpoint から Graph 用 access token を取得し、Microsoft Graph API 経由で SharePoint データを取得する
Confluence okta OAuth 2.0 Client Credentials（Okta） Cloud Run が Okta token endpoint で access token を取得し、Bearer token で Confluence REST API を呼び出す

3-2. 取得時と検索時の権限評価の分離

フェーズ 実行主体 何をするか 判定基準
取得（ETL） Cloud Run（Connector / ETL） 5システムへ接続し、データ本体と ACL 情報を取得・整形・格納する システム連携用のサービス認証（Okta / Entra ID / Okta LDAP Agent）
検索（Query） Gemini Enterprise + Agent Search 格納済みデータを検索し、ACL メタデータで閲覧可否を絞り込んで返却する。コンプライアンス/GDPR 区分のみ Cloud Run 側で二重チェックを追加する 一次判定: ユーザの SSO コンテキストとデータストア内 ACL。二次判定: Cloud Run のポリシーチェック（対象区分のみ）

どの目的で JWT 属性と ACL を突合するか、どのキーで判定するか、エビデンスを含む詳細は Agent Search 認可判定の詳細設計 を参照。

4. 監査

  • 検索要求
  • 参照したソース
  • 返却したドキュメント識別子
  • 失敗理由と処理時間

監査ログは後から追跡可能で、検索者・時刻・対象・結果を追えることを前提にする。

5. トレードオフ

  • 単一インデックス化よりもアクセス制御の正確性を優先する
  • ソース毎の Connector 分離で保守性を高める
  • 検索速度は十分な範囲で最適化し、まず安全性を担保する

6. 実装ロードマップ

  1. 公開用の検索 UI と runtime dashboard を維持する
  2. ソース別 Connector を実装し、メタデータとACLを取り込む
  3. 検索前にユーザー属性と ACL を突合する
  4. 監査ログと運用メトリクスを追加する

7. 今回の設計変更と反映先

設計変更 反映内容 反映先
Connector を ETL（Extract / Transform / Load）で分割 各責務を独立モジュール化し、順次実行できるパイプラインを導入 app/etl/extract.py, app/etl/transform.py, app/etl/load.py, app/etl/pipeline.py
5システム個別接続 ServiceNow / Workday / Compliance System / SharePoint / Confluence を個別設定で管理 .env.example, app/config.py
認証方式をシステム別に切替 Okta / Okta LDAP Agent / Entra ID を identity_provider で選択可能にした app/config.py, app/etl/extract.py
Transform で Document 化 + ACL 付与 取得データを Discovery Engine 向け文書へ正規化し、allowed_* メタデータを付与 app/etl/transform.py
Load の統一 Discovery Engine API 経由で BigQuery 宛先へロードする形へ統一 app/etl/load.py, app/config.py
デモ時は外部実接続しない ETL mock mode で 5システム取得、Secret 参照、LDAP bind 取得を固定レスポンス化 app/etl/mock_data.py, app/etl/extract.py, app/etl/load.py, app/main.py
