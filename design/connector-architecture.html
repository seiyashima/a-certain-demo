Connector Architecture

連携アーキテクチャ設計

Gemini Enterprise Agent Platform から呼ばれる Cloud Run アプリの公開可能な設計判断を整理したドキュメントです。

1. 設計ゴール

  • 複数ソースを統一的な検索体験で束ねる
  • 元システムの権限を維持し、見える情報だけ返す
  • 短命認証と監査ログを前提に運用する
  • Cloud Run 上で再現可能な構成にする
  • Search API / Ops UI / Connector Worker の責務を分離する

2. 主要コンポーネント

UI layer 検索画面と稼働確認画面を提供する
API layer /healthz, /api/runtime, /api/chat を提供する
Connector layer 各ソースの取得ロジックを個別実装で分離する
Identity layer Okta などの外部IdPと GCP の短命認証を仲介する

3. 認証・権限の考え方

ユーザーは既存SSOで認証し、そのアイデンティティをそのままソースアクセスの判断材料として扱う。 サービス側は長期資格情報を保持せず、必要な時だけ短命トークンを取得する。

  • ユーザー認証は既存IdPを起点に統一する
  • Connector はソース固有の権限モデルを保持する
  • 検索時には ACL フィルタで結果を絞り込む
  • ドキュメント更新時は再インデックスまたは削除を行う
  • Cloud Run サービスは責務別に分割し、独立して運用する

4. 監査

  • 検索要求
  • 参照したソース
  • 返却したドキュメント識別子
  • 失敗理由と処理時間

監査ログは後から追跡可能で、検索者・時刻・対象・結果を追えることを前提にする。

5. トレードオフ

  • 単一インデックス化よりもアクセス制御の正確性を優先する
  • ソース毎の Connector 分離で保守性を高める
  • 検索速度は十分な範囲で最適化し、まず安全性を担保する

6. 実装ロードマップ

  1. 公開用の検索 UI と runtime dashboard を維持する
  2. ソース別 Connector を実装し、メタデータとACLを取り込む
  3. 検索前にユーザー属性と ACL を突合する
  4. 監査ログと運用メトリクスを追加する
