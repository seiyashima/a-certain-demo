System Overview

Cloud Run 稼働アプリ 概要設計

対象: Gemini Enterprise Agent Platform 連携の呼び出し先として稼働する Flask アプリ

1. 目的

Cloud Run 上で API と確認画面を提供し、Gemini Enterprise からの呼び出し経路と、 人手での稼働確認経路を同時に確保する。

稼働確認画面では health/runtime/chat の3点を即時判定できることを重視する。

Cloud Run 上のアプリは Search API と Ops UI を責務分離し、必要に応じて同期用 Worker を別サービスとして運用する。

2. 非機能要件

  • Cloud Run の自動スケールに追従できるステートレス構成
  • Request ID ベースのログ相関
  • 環境変数による挙動制御
  • 障害時に UI と API の両面で切り分け可能

3. コンポーネント構成

コンポーネント 役割
Flask App Cloud Run で動くアプリ本体。エンドポイント提供、例外ハンドリング、ログ出力
/healthz 基本的な liveness/readiness 判定
/api/runtime Cloud Run 実行情報と稼働状況の可視化
/api/chat Gemini Enterprise 側呼び出しの入口（現時点は demo echo）
Verification UI (/) 人手での稼働確認画面。health/runtime/chat をワンクリック検証

4. 代表フロー

4-1. 人手による稼働確認

  1. 利用者がトップページへアクセス
  2. 画面が /healthz と /api/runtime をチェック
  3. 必要に応じて /api/chat へテストリクエスト送信
  4. 結果を OK/FAILED とメタ情報で表示

4-2. Gemini Enterprise 呼び出し

  1. Gemini Enterprise が API エンドポイントへリクエスト
  2. Flask が入力検証・処理実行
  3. レスポンス返却と request_id ベースのログ記録

5. API 契約（初版）

GET /healthz {"status": "ok"}
GET /api/runtime status, service, revision, project, environment, demo_mode, uptime_ms
POST /api/chat request body: {"message": "..."} / response: reply, request_id, elapsed_ms

6. 今後の拡張候補

  • Gemini Enterprise 実接続向けの認証方式・権限制御設計
  • 監査向けアクセスログとメトリクス収集の標準化
  • 疎通確認画面のチェック履歴保存とアラート連携

関連資料
