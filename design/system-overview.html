System Overview

Cloud Run 検索基盤 概要設計

対象: ACL チェックと検索クエリ配送を共通化した Cloud Run 上の検索ゲートウェイ

1. 目的

Cloud Run 上で共通の投げ口を提供し、ACL 判定・検索クエリ正規化・監査ログを 1 つにまとめたうえで、各検索システムへ安全に振り分ける。

共通部品は「ACL チェック」「クエリ整形」「ルーティング」「監査」の 4 つに分ける。

各システム固有の違いはアダプタ側に閉じ込め、共通層は入力と出力の形を固定する。

2. 非機能要件

  • Cloud Run の自動スケールに追従できるステートレス構成
  • Request ID ベースのログ相関と監査証跡の保存
  • ACL 失敗時は検索実行前に必ず遮断する
  • タイムアウト、リトライ、エラー形式を共通化する

4. 代表フロー

4-1. 共通検索フロー

  1. クライアントが共通の投げ口へ検索要求を送る
  2. 共通層がユーザ属性と格納済み ACL メタデータを照合し、通過した検索条件を正規化する
  3. コンプライアンス/GDPR 区分を含む結果のみ、Cloud Run 側で二重チェックする
  4. 対象システムを決めてアダプタにルーティングする
  5. 結果を共通スキーマへ整形して返却する
検索時の可否判定は、ETL で取り込んでデータストアに保存したデータと ACL メタデータを基準に行う。 元システムへの都度問い合わせではなく、格納済み情報を基準に表示可否が決まる。 標準は検索層で一次判定し、高機密区分のみ Cloud Run 側で二次判定する。

突合の目的、判定キー、失敗時の fail closed 方針、標準仕様への根拠は Agent Search 認可判定の詳細設計 に集約する。

4-2. 失敗時の扱い

  1. ACL 失敗は検索実行前に遮断する
  2. 外部 API 失敗は共通エラーコードへ変換する
  3. 監査ログには判定理由と trace を残す

5. 共通インターフェース（初版）

入力 subject, query, target_system, acl_context, correlation_id
出力 status, documents, matched_acl, normalized_error, trace
共通ルール ACL 失敗時は検索せず、各システムのレスポンスは共通形式へ正規化する

6. 非機能・セキュリティ・監視

  • Cloud Run の自動スケールに追従できるステートレス構成と短時間起動を前提にする
  • 長期資格情報を持たず、WIF と Secret Manager に接続情報を寄せる
  • health / runtime / search / connectors の疎通確認、成功・失敗観測、処理時間記録を運用指標にする
  • デモ用途では ETL mock mode を有効にし、5システム取得と Secret/BIND 参照を固定レスポンスで再現する

6-1. WIF と Secret Manager の棲み分け

この設計での役割分担は、WIF は外部サービスへ短命トークンで接続する仕組み、 Secret Manager はどうしても残る固定シークレットを保管する場所として分ける。

項目 WIF Secret Manager
主な役割 Cloud Run の実行主体を、外部IdPや外部SaaSが受け入れる短命認証情報へ変換する Client Secret、API Token、例外的な LDAP bind password などの固定シークレットを保管する
扱うもの トークン交換、フェデレーション、短時間だけ有効な access token 長めに保持される資格情報、接続先が WIF 非対応な場合の代替認証情報
保持方針 Cloud Run 側に固定秘密を持たせないことを優先する アプリコードや平文環境変数に置かず、参照時だけ取り出す
向いているケース Google Cloud から外部クラウドやSaaSへ、実行主体ベースで安全に接続したい場合 外部APIが client secret や bearer token を要求し、短命トークン化できない場合

6-2. 利用プロセス

  1. 基本は WIF（Federated Credential）を前提にし、Cloud Run のサービスアカウントから短命トークンを取得する
  2. 接続先が固定 secret を要求する場合のみ、Secret Manager から必要な secret を実行時に取得する
  3. 取得した認証情報で API を呼び出し、トークンや secret 自体はログへ出さない
  4. WIF で代替できる接続先が増えたら、Secret Manager に置いていた固定 secret を順次削減する
原則は「認証は WIF、保管は Secret Manager」。つまり、認証方式を WIF へ寄せ、完全に消せない固定秘密だけを Secret Manager に残す。
